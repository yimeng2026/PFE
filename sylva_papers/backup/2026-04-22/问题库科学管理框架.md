## 二、科学排序算法

### 2.1 优先级计算

```
PriorityScore = (Impact × Urgency) / (Effort + Risk)

其中:
- Impact:  层间辐射压力系数 (1-10)
- Urgency: 阻塞后续工作的程度 (1-10)
- Effort:  预计工时 (小时)
- Risk:    失败概率 (0-1)
```

### 2.2 依赖拓扑排序

```
问题A依赖问题B → B必须在A之前完成

当前依赖图:
Basic (基础层)
    │
    ├── NavierStokes ✅ (P0完成)
    ├── BSD (P2: 占位符→实体)
    ├── Hodge (P2: 占位符→实体)
    ├── Complexity (P1: Cook-Levin 7 sorry)
    │       └── CookLevin (P1: 核心证明)
    ├── NumericalZeros (P1: ZetaVerifier 1 sorry)
    │       └── ZetaVerifier (P1: 核心证明)

    └── SylvaInfrastructure (P1: CP004 1 sorry)
            └── CP004 (P1: 核心证明)
```

### 2.3 工作节奏

```
每日循环:
  08:00 检查问题库状态更新
  08:30 执行当日最高优先级任务
  12:00 中期检查点验证
  14:00 继续执行或调整优先级
  18:00 日终报告生成
  20:00 问题库更新 + 明日计划

每周循环:
  周一: 周计划制定 + 问题库全面审查
  周三: 中期检查点 + 优先级重排序
  周五: 周终报告 + 问题库归档
  周末: 深度工作（无中断的长时段任务）
```

## 三、科学执行规范

### 3.1 执行前检查清单

```
□ 问题是否已精确定义？
□ 依赖是否已满足？
□ 失败降级路径是否明确？
□ 输出标准是否清晰？
□ 验证方式是否可行？
```

### 3.2 执行中记录规范

```
每个任务必须记录:
- 开始时间
- 执行步骤（编号、操作、结果）
- 遇到的问题及解决方案
- 实际工时
- 输出文件位置
- 验证结果
```

### 3.3 执行后归档规范

```
任务完成 → 更新问题库状态 → 生成报告 → 更新依赖图 → 触发下游任务
```

## 六、自动化规则

### 6.1 触发条件

| 触发条件 | 自动行为 |
|---------|---------|
| 用户说「继续」 | 执行问题库中最高优先级待执行任务 |
| 用户说「深化改革」 | 启动7个并行Agent，贪婪全面执行 |
| 编译错误出现 | 自动触发截肢降级，记录到P0 |
| 任务完成 | 自动更新问题库状态，触发下游任务 |
| 每日08:00 | 自动生成问题库状态报告 |

### 6.2 禁止行为

- 未经排序直接执行低优先级任务
- 跳过依赖执行下游任务
- 不记录执行过程
- 不更新问题库状态
- 重复问用户已明确的问题
